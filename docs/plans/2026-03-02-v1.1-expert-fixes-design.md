## Mission

Fix all 5 issues identified by two independent expert reviews, and lay the
architectural groundwork for the OpenSage research lane via policy traits.

Two independent experts reviewed the v1.0.0 codebase. Their claims were
fact-checked against the code. Results:

## What Changes

### 1. Stop Finalization (P0)

**Problem**: TopologyPlanner generates an ExecutionPlan at UserPromptSubmit
that may omit finalization stages (calibrate, compact, decide, learn).
The Stop handler replays this plan instead of always finalizing.

**Fix**: Introduce `FinalizationPlan` as a separate concept. Stop handler
always runs `run_finalization()` which executes stages 9-12 regardless
of the topology plan stored at prompt time.

- New: `FinalizationPlan` struct in topology.rs
- New: `ControlLoop::run_finalization()` in runner.rs
- Change: Stop handler calls `run_finalization()` instead of `run_plan()`

### 2. Forge Tier 2 Fix (P1)

**Problem**: `after_post_tool_use()` passes `tool_response` (hook message)
to the syntax-checker Forge, but it should validate the actual file written
to disk.

**Fix**: Read the file from disk via `file_path` using `tokio::fs::read_to_string`.
If file is unreadable, skip silently (fire-and-forget context).

### 3. tool_need Real Heuristics (P2)

**Problem**: ToolNeedStage just sets `tool_necessary = tool_name.is_some()`.
No actual necessity assessment.

**Fix**: Add lightweight heuristics:
- Detect echo/printf-only Bash commands (no side-effects)
- Annotate `tool_necessity_reason` in LoopContext for advisory output
- Never block (advisory only, consistent with daemon's non-override model)

### 4. Memory Ranking via UCB (P2)

**Problem**: UnifiedSearch uses positional scores (1.0, 0.99, 0.98... for
events; 0.5, 0.49... for nodes). UCB adaptive_recall exists but is unused.

**Fix**:
- Expose real FTS5 BM25 rank scores from SQLite
- Wire UnifiedSearch to use adaptive_recall() when embeddings available
- Normalize scores from both sources to [0, 1] before merge
- UnifiedSearch receives EmbeddingProvider reference

### 5. State Drift Mitigation (P2)

**Problem**: Daemon accumulates rich SessionContext but Claude only sees
text strings in additionalContext. No structured query mechanism.

**Fix**: Add `GET /session/{session_id}/state` endpoint that serializes
SessionContext fields as JSON (task_type, risk, strategy, fatigue_score,
lessons, verification_results, tool_calls, errors, etc.).

### 6. Policy Traits (OpenSage Prep)

**Problem**: Expert 1 recommends separating stable core from research lane
via explicit policy interfaces.

**Fix**: Define three traits in core/src/policy.rs:
- `TopologyPolicy` — plan execution topology
- `MemoryPolicy` — persistence and ranking decisions
- `ToolSynthesisPolicy` — forge admission control

Current TopologyPlanner implements TopologyPolicy. No behavioral change,
but the trait enables pluggable OpenSage implementations later.

## What Does NOT Change

- No new hooks
- No changes to the hook protocol (HookInput/HookOutput schemas)
- No external LLM dependencies
- No changes to the plugin structure
- Daemon remains advisory + guard (no override)

## Success Criteria

1. Stop handler always runs calibrate/compact/decide/learn for all topologies
2. Forge Tier 2 validates actual file content, not hook response text
3. tool_need detects at least echo-only commands as unnecessary
4. Memory search uses real BM25 scores and UCB when available
5. GET /session/{id}/state returns structured JSON
6. TopologyPlanner satisfies TopologyPolicy trait
7. All existing 294 tests pass + new tests for each fix
8. cargo clippy clean, pnpm typecheck clean
